## Sistema de gerenciamento de filas

Apresentação do problema

Criar o back-end de uma aplicação para gerenciamentos de filas de atendimento (guichês de serviços). Essa aplicação será desenvolvida em Go. No primeiro momento não terá integração com banco de dados, tendo sua persistência de estado em memória volátil.

O sistema deve contar com o registro de profissionais que prestam determinado serviço. Os atendimentos podem iniciar ou encerrar a qualquer momento, desde que exista no mínimo um profissional atendendo naquele serviço ou não tenham mais clientes para serem atendidos no mesmo.

Um cliente deve emitir ficha de atendimento para si para que possa ser atendido. Esses clientes já são previamente classificados em categorias de preferência de atendimento, de modo que ao emitirem suas fichas de atendimento eles informam as suas categorias. As possíveis categorias são: atendimento imediato, prioridade média, baixa prioridade; diminuindo o grau de prioridade respectivamente.

O fim de um atendimento é registrado pelo profissional através da chamada do próximo cliente daquele(ficha) serviço. Um catálogo de serviços é criado previamente, de modo que ao iniciar seu expediente o atendente seleciona o seu serviço dentre os disponíveis.

A cada vez que uma ficha é chamada o sistema notificará via webhook um serviço externo com os dados do atendimento. Esse serviço deve ser configurado com o end-point que receberá o webhook.

Alguns dados do atendimento são: nome do serviço, nome do cliente, nome do profissional, categoria de atendimento, código da ficha, momento de emissão, momento de chamada.

O sistema deve fornecer uma API REST. Todos os end-points devem ser documentados em swagger. O sistema deve estar instrumentado com observabilidade via Open Telemetry. Todo o ambiente deve ser disponibilizado localmente com Docker/Docker compose.
Pode-se utilizar a stack de observabilidade:
- Grafana Tempo para traces
- Grafana Loki para logs
- Pometheus para métricas
- Grafana para visualização

## Domain - regra de negocio
deve conter:
- Entidades
- value objects
- regra de negocios puras
- erros de dominio

## Repository - interface de acesso a dados
deve conter: apenas interfaces nunca implementações

## Service - caso de uso da aplicação
ele orquestra:
 - Entidades do dominio
 - Repositories
 - mensageria
 - regras de fluxo

## extensions/adapters - implementações das interfaces
Aqui ficam as "traduções" entre o mundo externo e o dominio
- Implementações dos Repositories(postgres, mongodb, redis)
- client http externos
- mensageria(rabbitmq, kafka)
- cache
- integrações com APIs


## gateways/transports - pontos de entrada da aplicação
esta camada recebe requisições, valida e chama o service
- http handlers
- GraphQL
- websocket
